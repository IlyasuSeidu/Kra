# Frontend Gap Inventory

## Purpose

This file records actual gaps found by comparing:

- Canonical inventory: `docs/05-design/frontend-screen-inventory.md`
- Existing specs: `docs/05-design/frontend-screen-specs/**`
- Current frontend contract files: `apps/web`, `apps/mobile`, and `apps/admin`
- Implemented backend contracts in `services/api` and `packages/shared`

Rules used for this scan:

- Screens or pages that are not built yet are backlog, not gaps.
- Specs that are not written yet are backlog, not gaps.
- Backend gaps must be confirmed against implemented API routes, schemas, or service code before they are recorded.

## What Is Not A Gap

| Item                                                                    | Reason                                                                                            |
| ----------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| Unbuilt public, mobile, ops, courier, station, driver, or admin screens | These are implementation backlog until an implementation contradicts inventory/specs.             |
| Unbuilt shared UI systems                                               | These are implementation backlog unless an existing shared component contradicts inventory/specs. |
| Specs that are not written                                              | These are documentation backlog unless an existing spec conflicts with inventory.                 |
| Backend routes that already exist                                       | Confirmed implemented routes are listed separately and must not be re-filed as gaps.              |

## Confirmed Implemented Backend Support

These were checked and should not be treated as backend gaps:

- Public tracking routes exist: `GET /v1/public/track/:trackingCode`, `POST /v1/public/track/:trackingCode/request-verification`, and `POST /v1/public/track/:trackingCode/verify-phone`.
- Delivery lifecycle routes exist: create/list/detail/timeline/cancel/intake/assign driver/accept run/dispatch/pickup/in transit/receive destination/assign final mile/accept final mile/out for delivery/failed attempt/proof assets/complete.
- Payment routes exist: initialize, verify, refund, settle refund, payment webhook ingest, and internal reconciliation.
- Issue routes exist: list/create/detail/escalate/resolve, including delivery ID filtering.
- Admin list/summary routes exist: overview, deliveries, stations, launch readiness, finance, payment reconciliation, pricing rules, users, audit events, outbound notifications, and webhook events.
- `PACKAGE_SCAN_MISMATCH` exists in the API error taxonomy.
- Idempotent mutation handling exists through the `idempotency-key` header.
- Handoff fallback metadata exists in lifecycle request schemas and handoff records.
